# Urban Terraform with Ansible

This project provisions AWS infrastructure with Terraform and orchestrates planning/apply steps through Ansible playbooks. It is built around a two-step workflow:

1. Generate and stash a Terraform plan.
2. Load and apply the stashed plan.

This approach is useful for controlled automation pipelines where plan and apply happen in separate jobs or approval stages.

## Repository Layout

- `terraform/main.tf`: core AWS infrastructure and Terraform backend/provider settings.
- `terraform/output.tf`: Terraform outputs (currently `public_ip`).
- `terraform/user_data.txt`: EC2 bootstrap script.
- `playbooks/tf_init_plan.yml`: initializes Terraform, creates a plan, and stashes it.
- `playbooks/tf_apply_plan.yml`: loads and applies a previously stashed plan.
- `collections/requirements.yml`: required Ansible collections.

## What Gets Created

The Terraform configuration currently defines:

- VPC, subnet, internet gateway, and route table association.
- Security group with ingress for `22`, `80`, `443`, and `5432`.
- EC2 instance (Fedora CoreOS AMI lookup, `t2.micro`, 40 GB root volume).
- Ansible inventory host entry using the `ansible/ansible` provider.

## Prerequisites

- Terraform `>= 1.2.0`
- Ansible (with collection support)
- AWS credentials available in your environment (profile, role, or environment variables)
- Access to clone this repository (`git_url_tf`)

Install Ansible collections:

```bash
ansible-galaxy collection install -r collections/requirements.yml
ansible-galaxy collection install amazon.aws
```

## Backend Pattern (dev/stage/prod)

Both playbooks include a backend map keyed by `tf_environment`:

- `dev` -> `aap-on-aws/dev/terraform.tfstate`
- `stage` -> `aap-on-aws/stage/terraform.tfstate`
- `prod` -> `aap-on-aws/prod/terraform.tfstate`

Current defaults in playbooks:

- Bucket: `brthomps-tfstate-use1`
- Region: `us-east-1`
- Default environment: `dev`

Pass the environment at runtime:

```bash
-e tf_environment=dev
```

## Required Runtime Variables

The playbooks expect:

- `git_url_tf`: repository URL to clone into `urban/`
- `tf_environment` (optional): `dev`, `stage`, or `prod` (defaults to `dev`)
- `stashed_plan`: generated by `tf_init_plan.yml` and consumed by `tf_apply_plan.yml`

## Typical Workflow

### 1) Initialize and Create Plan

```bash
ansible-playbook playbooks/tf_init_plan.yml \
  -e git_url_tf=https://github.com/<org>/urban-terraform.git \
  -e tf_environment=dev
```

What this does:

- Ensures the backend S3 bucket exists.
- Clones the repo into `urban/`.
- Runs Terraform init/plan via `cloud.terraform.terraform` (`check_mode: yes`).
- Stashes the generated plan into the variable `stashed_plan`.

### 2) Apply the Stashed Plan

```bash
ansible-playbook playbooks/tf_apply_plan.yml \
  -e git_url_tf=https://github.com/<org>/urban-terraform.git \
  -e tf_environment=dev \
  -e @stashed_plan.json
```

`stashed_plan.json` should contain the `stashed_plan` value from the previous step.

## AWS Authentication

Do not store AWS access keys in Terraform files. Prefer one of:

- IAM role (best for automation runners)
- `AWS_PROFILE`
- `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` (and `AWS_SESSION_TOKEN` when needed)

## Troubleshooting

- **Collection/module not found**
  - Install required collections (`cloud.terraform`, `amazon.aws`).
- **Undefined `stashed_plan` in apply**
  - Ensure `tf_init_plan.yml` completed successfully and the stashed value is passed to apply.
- **Backend mismatch errors**
  - Use the same `tf_environment` for both plan and apply.
- **Implicit localhost warnings**
  - Expected for local-only runs with `hosts: localhost`.

## Security Notes

- The security group is currently open to the world (`0.0.0.0/0`) for several ports; tighten this before production use.
- `terraform/main.tf` currently contains an inline SSH public key; consider externalizing key material and using variables.
- The user-data script starts a simple PHP server on port `80`; replace with hardened service configuration for production.
